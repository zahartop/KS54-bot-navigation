# Production readiness audit (May 2026)

## Структура после реструктуризации

| Область | Путь |
|--------|------|
| Абитуриент (ДОД, специальности, главное меню, напоминания ДОД) | `src/logic/abi/` |
| Админ (заявки, CSV, рассылки, recovery) | `src/logic/admin/` |
| Модели и репозитории | `src/data/` |

`src/services/` сохранён для точки сборки AsyncIO-планировщика (PostgreSQL job store), не смешиваясь с доменными сценариями.

---

## 1. Race conditions (100+ одновременных пользователей)

**Наблюдение:** финальное сохранение ДОД/специальности идёт через `UserRepository._transactional_writes_with_retry()` с одной транзакцией на запись — для PostgreSQL это нормально: конкурирующие транзакции сериализуются блокировками строк/таблицы.

**Зона риска (TOCTOU):** перед записью ДОД вызывается `is_open_day_duplicate()`, затем в отдельной транзакции — `transactional_register_open_day()`. Между проверкой и вставкой теоретически возможна вторая заявка с тем же `telegram_user_id` и датой (два быстрых сабмита). От дублей частично защищают UX и троттлинг, но **гарантии на уровне БД нет**, если не добавить уникальный индекс вида `(telegram_user_id, date)` для `open_day_applications`.

**Специальности:** используется `is_specialty_duplicate` по одному пользователю — аналогичный класс гонки при параллельных сабмитах без уникального ограничения «одна активная заявка на пользователя».

**Рекомендация:** добавить ограничения уникальности в миграциях там, где бизнес-правило допускает не более одной записи.

## 2. Утечки памяти

- **Троттлинг:** `ThrottlingMiddleware` использует `TTLCache(maxsize=10000)` — верхняя граница ключей ограничена.
- **Рассылка:** список получателей — кортеж в памяти на время задачи; после завершения ссылку снимает GC. При очень больших базах возможен пик RSS — контролируется батчами/пагинацией на будущее.
- **APScheduler:** job store в БД — накопления джоб без лимита не в коде бота как утечка Python-объектов; мониторинг размера `apscheduler_jobs` — операционная задача.
- **ДОД напоминания:** `execute_dod_reminder` создаёт временный `Bot` и закрывает сессию в `finally` — утечка httpx-сессии маловероятна.

---

## 3. Security

- **Админ-действия:** проверки через `user_is_bot_admin` (один `ADMIN_ID`). Нет RBAC для нескольких админов — для мульти-тенанта колледжей потребуется список ролей или отдельная таблица.
- **Инъекции SQL:** запросы через SQLAlchemy ORM/параметризованные выражения — классической SQL-injection по пользовательской строке в сырой SQL из хендлеров нет.
- **Telegram HTML:** сообщения пользователей в рассылке с `parse_mode="HTML"` могут содержать теги; доверенный контент только от администратора — приемлемо при одном ADMIN.
- **Секреты:** токены и пароли через `.env` / pydantic settings — без вывода в логи при использовании PII-маскирования там, где это применяется к логгерам.

---

## 4. Error handling

- Глобальный обработчик `dp.errors` в `main.py` логирует и возвращает `True`, чтобы не ронять цикл опроса на одном плохом апдейте.
- `safe_handler` глушит любые `Exception` в декорированных хендлерах — разумно для UX, но **скрывает** редкие системные ошибки; критический путь (старт polling) обёрнут в `supervised_main` с перезапуском сессии.
- Репозиторий: узкий повтор транзакций по `SQLAlchemyError` и `TimeoutError`; общий `except Exception` один раз помечает сбой без ретрая — допустимо.

Широкие `except Exception: pass` встречаются в вспомогательных местах (напр. удаление напоминаний) — малорисково.

---

## 5. CTO: вердикт и масштабирование на 5–10 колледжей

**Текущее состояние:** один процесс бота, один tenant в конфиге (`content.py`, один `BOT_TOKEN`, один `ADMIN_ID`, одна БД). Перенос папок (`logic/abi` vs `logic/admin`) **улучшает навигацию**, но сам по себе не даёт multi-tenant.

**Готовность к 5–10 колледжам без доработок:** **низкая**. Нужны отдельно: мульти-instancing (Compose/K8s реплик по клиентам), параметризация текстов и брендов, возможно несколько ADMIN_ID или CRM-роли, общая платформа наблюдений.

**Оценка нагрузки (i9-12900k, 32 GB RAM, Docker):** один инстанс бота — лёгкая нагрузка на CPU/RAM для типичной приёмной комиссии Telegram (ограничение — **Webhook/getUpdates Telegram ~30 сообщ/сек на бота** и пул соединений к PostgreSQL `DB_POOL_SIZE` + overflow). При пиках ввода анкет 100 параллельных пользователей узкое место — **БД** и очередь записей, а не железо. Грубо: **десятки запросов/сек** к API Telegram и **десятки–сотни коротких транзакций/сек** к Postgres укладываются в один контейнер бота при нормальном Postgres на том же хосте. Точная цифра зависит от логов и `pg_stat`.

## 6. Три шага после запуска

1. **Уникальные ограничения и идемпотентность записи заявок** (снимают гонки при двойном сабмите).
2. **Наблюдаемость:** метрики RPS/handlers latency, трассировка медленных SQL, алерт на очередь `apscheduler_jobs` и размер очередей рассылки.
3. **Мульти-tenant roadmap:** конфиг клиента вне монолита `content.py` (БД/key-per-tenant), отдельные боты или единый с routing по `tenant_id`.

## Миграция APScheduler после смены пути jobs

Строки в job store могли ссылаться на `src.services.dod_reminders:execute_dod_reminder`. Теперь: **`src.logic.abi.dod_reminders:execute_dod_reminder`**.

После деплоя проверить таблицу job store и при необходимости выполнить SQL-апдейт `job_id` содержимого или перепланировать напоминания.
