## 📌 About · Зачем этот продукт

| ❌ Боль приёмной | ✅ Что даёт College Navigator Pro |
|------------------|-----------------------------------|
| Ручные переписки, потерянные контакты, хаос в Excel и чатах | **Структурированные заявки** сразу в базе |

## ⭐ Core Features

| Область | Описание |
|---------|----------|
| 🔐 **Middleware ПДн** | Глобальная проверка статуса согласия до выполнения сценариев с записью и анкетами |
| 🧱 **Repository pattern** | Изоляция доступа к БД — проще тестировать, менять схему и аудировать операции |
| ⚡ **Async-first** | `asyncio` + асинхронный PostgreSQL (`asyncpg` / SQLAlchemy 2) — высокая отзывчивость под нагрузкой |
| 🐳 **Docker-инфраструктура** | Compose-сервисы **PostgreSQL**, **Redis** (FSM), бот с health-aware зависимостями |
| 🔁 **Сеть и перезапуски** | Настраиваемые таймауты к Telegram API, ретраи при старте, контролируемый supervisor-сценарий |
| 📅 **Фоновые задачи** | APScheduler: напоминания о ДОД и мониторинг (watchdog) без блокировки обработки апдейтов |
| 📈 **Встроенный мониторинг** | Healthcheck-эндпоинты, контроль латентности БД и Telegram-алерты администраторам |
| 📜 **Миграции** | Alembic — воспроизводимая эволюция схемы от dev до prod |

> 💡 **Акцент:** архитектура заточена под **predictable operations** — наблюдаемость в логах, ограничение blast radius при сбоях провайдера и понятный контур восстановления.

## 🏗 Architecture

Проект реализован как **modular monolith**: один деплоибельный процесс (бот), внутри — **чёткое разделение модулей** без распределённой связности микросервисов.

```text
┌─────────────────────────────────────────────────────────┐
│  Telegram updates → Middleware → Routers / Handlers     │
└─────────────────────────────┬───────────────────────────┘
                              │
┌─────────────────────────────▼───────────────────────────┐
│  Application logic (сценарии ДОД, специальности, админ)  │
└─────────────────────────────┬───────────────────────────┘
                              │
┌─────────────────────────────▼───────────────────────────┐
│  Repositories · доменные операции над данными            │
└─────────────────────────────┬───────────────────────────┘
                              │
┌─────────────────────────────▼───────────────────────────┐
│  Infrastructure: PostgreSQL · Redis · Scheduler · Logs    │
└─────────────────────────────────────────────────────────┘
```

**Почему это круто для продукта колледжа:**

- 🎯 **Один контур деплоя** — ниже операционная стоимость, проще сопровождение ИТ-отдела колледжа или подрядчика.
- 📦 **Модули с явными границами** — можно без боли расширять сценарии (новые формы, отчёты, интеграции с LMS).
- 🔒 **Единая политика безопасности** — middleware и аудит проходят через один конвейер запросов.

## 🚀 Quick Start

### Требования

- установленные **Docker** и **Docker Compose**
- файл переменных окружения **`.env`** в корне проекта (рядом с `docker-compose.yml`)

### 1. Переменные окружения

```bash
cp .env.example .env
# Отредактируйте .env: BOT_TOKEN, POSTGRES_*, при необходимости REDIS_HOST и др.
```

### 2. Запуск стека

```bash
docker compose build
docker compose up -d
```

Сервис **бота** стартует после того, как **PostgreSQL** и **Redis** перейдут в состояние **healthy**; контейнер бота настроен на **`restart: unless-stopped`**.

### 3. Миграции и логи

При типичном деплое миграции Alembic выполняются при старте контейнера (**`docker-entrypoint.sh`**). Подробности — в **[`DEPLOY.md`](./DEPLOY.md)**:

```bash
docker compose logs -f bot
docker compose exec bot alembic upgrade head   # при необходимости вручную
```

## 🛡 Safety & Security · ПДн и комплаенс

| Принцип | Реализация в продукте |
|---------|----------------------|
| **Lawfulness / законность** | Сбор чувствительных данных только после **экрана согласия** и учёт статуса в БД |
| **Purpose limitation** | Middleware ограничивает функциональность **до принятия политики** для сценариев записи и анкет |
| **Transparency** | Пользователю показывается политика и понятный текст перед решением «согласен / отказ» |
| **Operational security** | Маскирование ПДн в логах, разделение prod/dev зависимостей (`requirements.txt` / `requirements-dev.txt`) |

Фокус на **российском режиме ПДн (ФЗ-152)** и **best practices**, близких к **GDPR** (минимизация данных, явное согласие, контроль точек входа в формы).
